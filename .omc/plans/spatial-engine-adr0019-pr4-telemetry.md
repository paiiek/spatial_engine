## §0 — RALPLAN-DR Summary

### Principles (P)
- **P1 — Audio thread is sacred (untouched by PR4).** `poll_diagnostics` and ALL OSC emission run on the control/IO thread (the standalone `while` loop). PR4 NEVER touches the RT `pump_block`/`thread_loop` path. The 4 counters are already RT-incremented (`underrun_warnings_` at `SharedRingBackend.cpp:417` after `record_underrun`); PR4 only READS them on the control thread via the `const noexcept` accessors. No new RT-path code, so the RT-sentinel lane is unaffected by the emitter (the emitter ctest is a control-thread test).
- **P2 — Edge-triggered, never per-tick spam — with the underrun 1/s guarantee emitter-owned (MJ-1 correction).** Emit `/sys/warning shm_*` once per counter INCREMENT (delta since the previous tick), NOT once per tick. **Three of the four counters are source-rate-limited inside `poll_diagnostics`** (attached-no-data once-on-attach latch `:505-510`; stale once / 30 s `:514-523`; pacing once / 5 s `:541-545`) — for those PR4 just maps "counter went up" → "emit the edge", adding NO second rate-limiter. **`shm_underrun` is the exception:** `underrun_warnings_` is `++`'d on the RT path at `SharedRingBackend.cpp:417` on EVERY silent block with NO source rate-limit (`poll_diagnostics` never touches it). ADR §6 (`:108`, `:223`) mandates `shm_underrun` at **once per second**. Therefore PR4's emitter OWNS the underrun 1/s guarantee via an emitter-side `last_underrun_emit_ms_` latch keyed on the injected `now_unix_ms` (cadence-INDEPENDENT: at most one `shm_underrun` per 1000 ms wall-clock window regardless of the diagnostics-tick rate). A single tick that sees deltas on multiple codes emits one packet per code that moved (D1); the underrun packet is additionally gated by the 1/s window.
- **P3 — Additive, zero PR2/PR3 regression.** All current ctests + RT-sentinel + pytest stay green. PR4 adds NO header field, NO new `BackendError`, NO new OSC typetag, and does NOT alter `poll_diagnostics`. The diagnostics tick is the ONLY change to `spatial_engine_core.cpp`'s loop and is `if (input_is_shm)`-gated, so the dante/null-output paths are byte-identical.
- **P4 — Wire compatibility.** `/sys/warning` reuses the frozen `,iis 0 0 "<code>" "<detail>"` shape (the canonical D-5 control-thread form at `SpatialEngine.cpp:1074`). `/sys/state` reuses the additive dual-tag `,s "key=value"` form (`docs/ipc_schema.md:336-366`; emit precedent `vst3/SpatialEngineProcessor.cpp:1365`), so a v0 `,i`-bitmask-filtering consumer is unaffected. NO new typetag, NO `schema_version` prefix on the `,s` form (per the ipc_schema dual-tag contract).
- **P5 — Don't break PR5/PR6/PR7 seams.** PR5's Python producer WRITES `producer_heartbeat_ms` (wall-clock ms, ADR §2.3 line 81) + `producer_state` + zero-inits `_reserved`; PR4 only READS. PR6 soak asserts these `/sys/warning shm_*` + `/sys/state` cross-process. PR7 is cross-platform + (if deferred) the D5 hardening. The clock-domain contract PR4 picks (D2) is a PR5 obligation the soak verifies.
- **P6 — Reuse the proven extract-and-test precedent.** The per-tick decision is factored into a `spe::bin::ShmTelemetryEmitter` header-only unit driven by an injected clock + injected `OSCBackend&`, EXACTLY as `PlayerStaleWatchdog.h`/`servicePlayerStaleWatchdog` factored the D-5 watchdog and got `test_phase_b_sync_handlers.cpp` case 11. The standalone loop calls the same entry point production drives — no untested inline `main()` logic.

### Decision Drivers (top 3)
- **DD1 — Edge-correctness without double-rate-limiting, AND no edge lost on no-peer.** The detection+rate-limit policy is FROZEN in `poll_diagnostics` (PR2) for 3 of 4 codes; `shm_underrun` has NO source rate-limit so the emitter OWNS its ADR §6 1/s guarantee (MJ-1). PR4 must emit exactly the new edges and nothing more — a per-tick emit would spam — yet a warning edge fired before an OSC peer connects must NOT be dropped (CR-2 retry-on-no-peer hold), because PR6's soak asserts these cross-process and can only do so if the edge is recoverable. The emitter is a delta function over monotonic counters with a per-code delivery latch → testable in isolation.
- **DD2 — Cross-process clock-domain correctness.** `poll_diagnostics` compares `now_ms - producer_heartbeat_ms`. If PR4 feeds the wrong clock (`steady` vs `unix-epoch`) the stale detector either never fires or always fires. ADR §2.3 line 81 pins `producer_heartbeat_ms` = "Wall-clock ms" → unix epoch. PR4 must feed `now_unix_ms` (system_clock) for `now_ms` and steady ns for `now_ns`, and record the contract as the PR5 producer obligation.
- **DD3 — Additive, JUCE-OFF, no-regression.** The tick + emitter must build under `-DSPATIAL_ENGINE_NO_JUCE=ON` (CI), keep all current ctests green, add no RT-path code, and reuse only existing `sendReply` overloads (`,iis` + `,s`) — no `OSCBackend` API change, no `RingHeader` change.

## Decision 1 — Edge-detection mechanism + per-code detail-string content

**The four counters (verified `SharedRingBackend.h:133-136`, all `unsigned long long`, `const noexcept`):**
`underrunWarningCount()` → `shm_underrun`; `staleWarningCount()` → `shm_producer_stale`; `pacingWarningCount()` → `shm_producer_pacing`; `attachedNoDataWarningCount()` → `shm_attached_no_data`. All four are monotonic non-decreasing (only `++` ever; verified — no reset path). `underrun_warnings_` is incremented on the RT path (`:417`) AND is otherwise touched only by reads; the other three are incremented only inside `poll_diagnostics` (control thread). So a control-thread reader sampling all four after a `poll_diagnostics` tick observes a consistent monotonic snapshot.

**Benign-race note (fold-in 7c, PR2-Q3 house style):** the control-thread emitter reads `underrun_warnings_` (a plain `unsigned long long`) while the RT thread `++`'s it (`:417`). This is a **benign-by-architecture data race** — on x86-64 / ARM64 an aligned 64-bit word load/store is atomic at the hardware level, so the reader sees either the pre- or post-increment value, never a torn word; a stale-by-one read self-corrects on the next 1 Hz tick (the counter is monotone and the detail carries the cumulative total). This matches the EXISTING `driver->xrunCount()` control-thread read at `spatial_engine_core.cpp:636-638` (the teardown printf) and the PR2-Q3 "no-syscall / benign-read" accounting style. PR4 does NOT promote `underrun_warnings_` to `std::atomic` (that would be an RT-path change → P1 violation surface) — it documents the read as benign, exactly as PR2 did for the same counter.

### Viable Options

| Option | Mechanism | Pros | Cons |
|---|---|---|---|
| **(a) Cache prev-4 in the emitter; diff each tick; emit one `/sys/warning` per code whose count increased.** Detail string = the relevant header value at emit time (age-ms for stale, count for underrun, etc.). | Pure function over monotonic counters; trivially unit-testable (feed counts, assert packets); matches the D-5 `,iis` precedent (sends a meaningful value, not a raw delta); no new state in the backend. | The emitter holds 4 prev-values + needs the header readable for the detail string (it has the backend ref). Bounded. |
| **(b) Coalesce all moved codes into ONE `/sys/warning` packet per tick.** | One packet/tick. | Breaks the frozen `,iis 0 0 "<code>" "<detail>"` shape (one code per packet); a consumer that switches on `<code>` would need re-parsing; no precedent. Rejected. |
| **(c) Push-model: have `poll_diagnostics` invoke a callback on each edge.** | Emit at the exact edge inside the backend. | Violates the FROZEN `poll_diagnostics` WRITES-set contract (`SharedRingBackend.h:138-147`) + would put an OSC-shaped dependency inside the RT-adjacent backend (it must stay OSC-free per `SharedRingBackend.h:52` "does NOT emit OSC"). Rejected — breaks P3 and the PR2 seam (PR2-Q1: PR4 is TRANSPORT, backend is POLICY). |

### Recommended: **(a) prev-4 cache + per-code edge emit, detail string = the meaningful header/count value.**

**Rationale.** The PR2 seam (PR2-Q1, FROZEN) is explicit: PR2 owns detection+rate-limit POLICY, PR4 owns TRANSPORT. Option (c) would re-merge them and break the `SharedRingBackend.h:52` "no OSC" contract; option (b) breaks the one-code-per-packet wire shape. Option (a) is a pure delta over monotonic counters — exactly what the `ShmTelemetryEmitter` unit can test in isolation by feeding synthetic counts. It mirrors the D-5 precedent which sent a *meaningful* value (`age_ms/1000` seconds) in the `,iis` detail string, not a raw delta.

**Per-code detail string (resolved against the D-5 precedent — `player_heartbeat_stale` sent age-in-seconds):**

| Code | When emitted | `<code>` (string arg 1) | `<detail>` (string arg 2) |
|---|---|---|---|
| `shm_underrun` | `underrunWarningCount()` increased **AND** the emitter-side 1/s window is open (MJ-1: `now_unix_ms - last_underrun_emit_ms_ >= 1000`) | `"shm_underrun"` | total xruns as a string: `"<xrunCount()>"` (the cumulative underrun count is the actionable operator value; matches ADR §6 "shm_underrun (1/s)" — the emitted detail is the running total, edge-triggered + 1/s-gated by the emitter) |
| `shm_producer_stale` | `staleWarningCount()` increased | `"shm_producer_stale"` | heartbeat age in seconds: `"<(now_unix_ms - producer_heartbeat_ms())/1000>"` (BYTE-FOR-BYTE the D-5 shape — `player_heartbeat_stale` sent `age_ms/1000`) |
| `shm_producer_pacing` | `pacingWarningCount()` increased | `"shm_producer_pacing"` | `"pacing_drift"` (a fixed marker; the pacing delta is internal to `poll_diagnostics` and not re-exposed — the operator-actionable fact is "producer pacing drifted", consistent with ADR §6 "shm_producer_pacing (1/5s)") |
| `shm_attached_no_data` | `attachedNoDataWarningCount()` increased (once, on attach) | `"shm_attached_no_data"` | `"channels=<inputChannelCount()>"` (matches ADR §6 "/sys/warning shm_attached_no_data once on attach" and §2.5 "shm_underrun \"<channels=…>\"" detail-style precedent) |

All four use `sendReply("/sys/warning", ",iis", 0, 0, "<code>", "<detail>")` (the existing overload, `OSCBackend.h:135-137`). The two leading ints stay `0, 0` exactly as the D-5 call (schema/type placeholders).

**Underrun 1/s latch (MJ-1):** the emitter holds `last_underrun_emit_ms_` (a `std::uint64_t`, init 0). On a tick where `underrunWarningCount() > prev_underrun_` AND `(now_unix_ms - last_underrun_emit_ms_) >= 1000` AND a peer is present, emit ONE `shm_underrun`, set `last_underrun_emit_ms_ = now_unix_ms`, and advance `prev_underrun_` to the count just sent (the cache-advance-on-sendReply-true rule, PM1). This is **cadence-independent**: it bounds emission to ≤1 per 1000 ms wall-clock window even if the diagnostics tick runs faster (D4 is 1 Hz today, but PR4-Q5's contract must survive a future bump). Two distinct suppressions compose without conflict: (1) the **1/s window** (`< 1000 ms` since last send) suppresses a NEW emit but leaves `prev_underrun_` BEHIND so the still-pending underrun is re-evaluated next window; (2) the **no-peer HOLD** (CR-2) likewise leaves `prev_underrun_` behind until `sendReply` confirms true. In both cases the single PM1 rule applies — `prev_underrun_` advances ONLY on a confirmed send — so a window- or peer-suppressed underrun is neither double-emitted nor lost. The other three codes have NO emitter-side window (their source rate-limits in `poll_diagnostics` are authoritative); they too follow the PM1 advance-on-send-true rule for the no-peer case.

**No-peer retry for `/sys/warning` edges (CR-2 — preferred option (a)).** `sendReply` returns false when no OSC peer has been captured yet (`OSCBackend.h:118-119`). A warning edge that fires before any UI/soak client connects (the realistic case — the standalone loop's existing one-shots wait `emit_after_ms`=1000 ms for exactly this peer-capture reason, `spatial_engine_core.cpp:567-569`) MUST NOT be dropped: PR6's soak (`test_phase_c_shm_loopback.py`) is contracted to assert these cross-process, which is only possible if the edge is recoverable. **Mechanism:** the emitter does NOT advance a code's prev-cache until that code's packet is `sendReply`-confirmed `true`. Per code it holds the pending edge (the prev-value stays BEHIND the current count, so the diff stays positive) and re-attempts `sendReply` each tick until it succeeds, then advances prev to the count observed at confirmation. For `shm_underrun` the 1/s window and the no-peer hold compose: the window gates *new* emits; the no-peer hold keeps the *un-delivered* edge armed (re-attempted next tick, still respecting the 1/s window from the last SUCCESSFUL send). This is the SAME retry-on-no-peer latch the `/sys/state ,s` path uses (`vst3/SpatialEngineProcessor.cpp:1362-1371`), now applied to `,iis` warnings too — closing the fire-and-forget gap that `SpatialEngine.cpp:1074` (D-5) left (D-5's player-stale warning IS droppable on no-peer; PR4's shm warnings are NOT, because PR6 asserts them).

**Consolidated seed + emit semantics (CR-2 — ONE explicit table, supersedes the scattered prose):**

| Signal class | First (seed) tick | Subsequent ticks | No-peer behavior |
|---|---|---|---|
| **`/sys/warning` (4 codes, D1)** | SEED-SILENT: capture current counts into prev-cache, emit NOTHING (a count that was already non-zero at first tick — e.g. an attached-no-data latch fired before the emitter's first tick — is captured as baseline, NOT replayed; D4 places the first tick immediately at loop entry so this window is ~0). | Emit on `count > prev` (per-code), underrun additionally 1/s-gated. | HOLD: prev-cache NOT advanced until `sendReply==true`; edge re-attempted each tick (CR-2 option a). |
| **`/sys/state` (3 keys, D3)** | SEED-AND-EMIT: capture current field values into prev-cache AND arm all three latches so the initial snapshot is emitted (a late-connecting consumer learns current state via the retry latch). | Emit on value-change (per-key); arm that key's latch. | HOLD: per-key latch stays armed until `sendReply==true` (the frozen `fallback_mode` semantics). |

The asymmetry is deliberate and now explicit: warnings seed-silent (a warning is an EVENT — replaying a pre-existing count as a fresh event would be a false alarm); state seeds-and-emits (state is a LEVEL — a late consumer must learn the current level). Both classes use the same no-peer HOLD so nothing is lost before a peer connects.

## Decision 2 — `now_ms` / `now_ns` clock-domain contract for `poll_diagnostics`

`poll_diagnostics(now_ms, now_ns)` (`SharedRingBackend.cpp:496`) uses:
- `now_ms` ONLY for staleness: `now_ms - producer_heartbeat_ms` (`:517`) and the 30 s / 5 s rate-limit windows (`:519`, `:541`).
- `now_ns` is currently `(void)now_ns` in PR2 (`:501`) — reserved for the PR4 OSC timestamp field; the pacing logic uses the HEADER's `producer_meta_block_pts_ns` deltas (`:529-536`), NOT `now_ns`.

**The contract to nail (DD2):**
- **`producer_heartbeat_ms`** — ADR §2.3 line 81: **"Wall-clock ms of last successful write. Stale > 100 ms = producer died."** → **unix epoch ms**. So `now_ms` MUST be `now_unix_ms` (the loop ALREADY computes it via `system_clock` at `spatial_engine_core.cpp:585-591`).
- **`producer_meta_block_pts_ns`** — ADR §4.3 line 194: stamped at producer-side **CLOCK_MONOTONIC ns** ("producer-monotonic-ns t0"). Pacing compares *successive header pts deltas* (both producer-clock values) — so the consumer's `now_ns` is NOT actually compared against the header pts (PR2 only diffs header-vs-header). `now_ns` is therefore the engine's `steady_clock` ns, carried for the PR4 OSC timestamp field only.

### Viable Options

| Option | `now_ms` source | Pros | Cons |
|---|---|---|---|
| **(a) `now_ms = now_unix_ms` (system_clock), `now_ns = steady_clock ns`.** | system_clock epoch ms (already in the loop) + steady ns | Matches ADR §2.3 "wall-clock ms"; reuses the EXISTING `now_unix_ms` the D-5 watchdog already computes (`:585-591`); stale detection correct cross-process IF PR5 writes wall-clock ms (a PR5 obligation, recorded as PR4-Q1). | Wall-clock is non-monotonic (NTP step) — a backward step could briefly mis-fire/suppress stale; bounded by the 30 s rate-limit + the >100 ms threshold (an NTP step large enough to matter is rare and self-corrects next tick). |
| **(b) `now_ms = steady_clock ms`.** | steady ms | Monotonic. | WRONG: `producer_heartbeat_ms` is wall-clock (different epoch + different process), so `now_steady_ms - producer_unix_ms` is a garbage difference (off by the unix epoch ≈ 1.7e12 ms) → stale ALWAYS fires. Rejected. |

### Recommended: **(a) `now_ms = now_unix_ms` (system_clock epoch ms), `now_ns = steady_clock ns`.**

**Rationale.** ADR §2.3 line 81 is unambiguous: `producer_heartbeat_ms` is **wall-clock ms**. The only correct consumer clock for the `now_ms - producer_heartbeat_ms` difference is the same wall-clock epoch — `now_unix_ms`, which the loop already computes for the D-5 watchdog (`:585-591`), so PR4 reuses it (one `system_clock` read per tick, control-thread, allocation-free). `now_ns` = `steady_clock::now().time_since_epoch().count()` (the loop's `now` is already `steady_clock`), passed through to satisfy the `poll_diagnostics` signature and reserved for the PR4-emitted OSC timestamp slot (currently the `,iis` leading ints are `0,0`; PR4 does NOT add a timestamp arg — the `0,0` shape is frozen — so `now_ns` remains a forward-compat carry, NOT consumed in PR4 emission). The NTP-step risk (option a con) is bounded by the existing 30 s rate-limit and the 100 ms threshold and is the SAME exposure the D-5 player-heartbeat watchdog already accepts (it also uses `now_unix_ms`). **The cross-process correctness of stale detection depends on PR5 writing `producer_heartbeat_ms` as unix-epoch ms — recorded as PR4-Q1 (a PR5 producer obligation the PR6 soak verifies).**

## Decision 3 — `/sys/state` shm fields: cadence + form + value encoding

ADR §6 adds three state fields: `shm_producer_alive` (0 when heartbeat stale), `shm_producer_state` (0=idle/1=streaming/2=draining/3=closed, from `producer_state()`), `shm_consumer_locked` (from the consumer-lock word at `kConsumerLockOffset`).

**Available sources (verified):** `producer_state()` (`SharedRingBackend.h:131`, returns the `ProducerState` u32); `producer_heartbeat_ms()` (`:130`) → derive `shm_producer_alive` (see derivation below); the lock word via the NEW `consumerLockWord() const` accessor (CR-1) — the emitter does NOT reach into `region_`/`consumer_lock_atomic` directly (both are private/non-const-unsafe), it calls the one additive backend accessor: `shm_consumer_locked = (consumerLockWord() != 0) ? 1 : 0` (boolean per v1).

### Viable Options

| Axis | Option | Pros | Cons |
|---|---|---|---|
| **Cadence** | **(c1) On-change (edge), with retry-on-no-peer latch** | Matches the FROZEN `,s` precedent exactly (`docs/ipc_schema.md:351` "Emitted once per lifetime … retry-on-no-peer"; `vst3/SpatialEngineProcessor.cpp:1360-1367` drains a pending latch each tick, keeps it armed until `sendReply` returns true). Low wire noise. | Slightly more state (prev-value cache + pending latch) than periodic. |
| | (c2) Periodic every tick / 1 Hz | Trivial. | Spams `/sys/state` (3 fields × 1 Hz indefinitely); the existing `,s` form is explicitly NOT periodic (the `,i` bitmask is the 10 Hz heartbeat). Diverges from the `,s` precedent. Rejected as default. |
| **Packet form** | **(f1) Three separate `,s "key=value"` packets** (`"shm_producer_alive=0"`, `"shm_producer_state=2"`, `"shm_consumer_locked=1"`) | EXACTLY the `fallback_mode=…` precedent (one `,s "key=value"` per snapshot); each field has its own pending-latch so a no-peer retry re-sends only what's unconfirmed; consumer dispatches per-key. | Up to 3 packets when all change. |
| | (f2) One coalesced `,s "alive=…;state=…;locked=…"` | One packet. | New key-list grammar (`;`-separated) not in the `,s` precedent (which is single `key=value`); a `fallback_mode` consumer parser would need extending. Rejected. |
| **`shm_consumer_locked` value** | **(v1) boolean 0/1** (locked vs unlocked) | The field NAME is `_locked` (boolean); the holder PID is an internal detail; matches the ADR §6 "shm_consumer_locked" naming. | PID not exposed (acceptable; PID-level diagnostics belong to a future field). |
| | (v2) holder PID | More info. | The lock word is THIS consumer's own PID once it acquires (always non-zero while running) → reporting the PID is near-constant and leaks the engine PID onto the wire for no operator value. `_locked` is the documented field. Rejected. |

### Recommended: **(c1) on-change + retry-on-no-peer latch, (f1) three separate `,s "key=value"` packets, (v1) boolean `shm_consumer_locked`.**

**Rationale.** The `,s "key=value"` form is the EXACT frozen precedent (`docs/ipc_schema.md:336-366`) — additive, typetag-dispatched, NOT periodic, drained with a retry-on-no-peer latch (the `sendReply` return value gates the latch, `vst3/SpatialEngineProcessor.cpp:1362-1367`). PR4 mirrors this per field: the `ShmTelemetryEmitter` holds a prev-value + a pending-latch per state field; on a value change it arms the latch; each tick it attempts `sendReply("/sys/state", ",s", "shm_producer_state=2")` and clears the latch only on success (so a no-peer tick keeps it armed — identical semantics to `fallback_mode`). Three separate packets keep each `key=value` atomic and self-describing (f2's `;`-grammar has no precedent and would break the `fallback_mode` consumer contract). `shm_consumer_locked` reports boolean 0/1 to honor the field NAME and avoid leaking the engine PID. **`shm_producer_alive` is derived** from the same `> kStaleThresholdMs` predicate the stale-warning uses, so `/sys/state.shm_producer_alive=0` and the `/sys/warning shm_producer_stale` edge are consistent by construction (ADR §6 line 224 pairs them).

**Emit timing:** the state-field evaluation happens on the SAME diagnostics tick as the warning edges (D4), AFTER `poll_diagnostics` (so the heartbeat read is the freshest). The first tick SEEDS the prev-state cache and emits the initial snapshot of all three (the seed-and-emit row of the CR-2 table).

**`shm_producer_alive` derivation + detached-guard (fold-in 7b).** `shm_producer_alive = (region attached AND (now_unix_ms - producer_heartbeat_ms()) <= kStaleThresholdMs) ? 1 : 0` (reuse the 100 ms threshold from `SharedRingBackend.h:77`). The attachment guard is load-bearing: `producer_heartbeat_ms()` returns **0** when `!region_.is_attached()` (`SharedRingBackend.cpp:351`), and `now_unix_ms - 0` is a huge value that would make `shm_producer_alive=0`/stale ALWAYS fire. **The emitter gates its ENTIRE tick on attachment** — the D4 tick is already `if (input_is_shm)`-guarded, and the backend stays attached for the engine's lifetime (the producer owns the shm lifecycle; the consumer never unmaps mid-run — `region_` is detached only in the dtor). So in normal operation `is_attached()` is true throughout the run. The emitter additionally early-returns from `tick()` when a (future) detached state is observed (defensive: a `consumerLockWord()`/`producer_heartbeat_ms()` of 0 from a detached region must not be misread as "producer stale" — the correct signal there is "no shm", not "producer died"). This keeps the `shm_producer_alive=0` ⇔ `shm_producer_stale` edge pairing honest (see AC sub-case below).

**`shm_producer_alive`/`shm_producer_stale` co-emit (fold-in 7a).** Because both derive from the SAME `> kStaleThresholdMs` predicate evaluated on the SAME tick (after `poll_diagnostics`), when the heartbeat crosses the 100 ms threshold the stale-warning counter edges AND `shm_producer_alive` flips to 0 on that tick — they co-emit. The "consistent by construction" claim is now AC-tested (AC-4 sub-case): one tick with a stale heartbeat must produce BOTH a `/sys/warning shm_producer_stale` edge AND a `/sys/state ,s "shm_producer_alive=0"` (ADR §6 line 224 pairs them).

**1 Hz `producer_state` coalescing bound (MJ-2).** The emitter reads `producer_state()` ONCE per tick (D4 = 1 Hz). A producer lifecycle transition SHORTER than the tick period — notably `Draining(2) → Closed(3)` if the producer drains and closes within one second — is **coalesced**: only the latest `producer_state()` value per tick is observed, so an intermediate `shm_producer_state=2` may never emit. ADR §6 (`:225-226`) treats Draining and Closed as DISTINCT operator states. This bound is accepted for PR4 and resolved by **PR4-Q6** (either PR5's producer holds each non-terminal state ≥ one tick period — a producer obligation like PR4-Q1 — OR PR6's soak asserts only the TERMINAL `Closed` state cross-process, treating intermediate-state observation as best-effort). PR4 does NOT add sub-tick state sampling (that would require a producer→consumer state-change signal absent from the wire — out of PR4 scope). Documented here so a future reader knows the 1 Hz tick under-samples fast transitions by design.

## Decision 4 — Diagnostics-tick placement + cadence (resolves PR3-Q4)

PR3 deferred the diagnostics tick (PR3-Q4): the standalone loop (`spatial_engine_core.cpp:582-631`) does NOT call `poll_diagnostics`. The loop already ticks the D-5 watchdog gated to ~1 Hz (`:585-594`), computes `now_unix_ms` (`:588-591`), and `sleep_for(50ms)` (`:630`). The shm handle is `shm_backend` (a `unique_ptr<SharedRingBackend>` under the linux/macOS guard); the emit channel is `engine.oscBackend()` (used for `/sys/binaural_warning` at `:624`).

**ADR §6 line 166:** "`producer_heartbeat_ms`/`producer_state` reads happen on the control thread … `HeartbeatPublisher` already runs there at 10 Hz." `poll_diagnostics`' internal rate-limits are 30 s (stale) / 5 s (pacing) / once-on-attach (no-data), so calling it at 1 Hz or 10 Hz is equally safe (the source rate-limits dominate).

### Viable Options

| Option | Placement / cadence | Pros | Cons |
|---|---|---|---|
| **(a) New `if (input_is_shm)` block right after the D-5 watchdog block, gated to ~1 Hz** via a `last_shm_diag_check` steady time_point (mirror `last_stale_check`). | Symmetric with the D-5 watchdog (same 1 Hz gate, same `now`/`now_unix_ms` already computed); reads `producer_heartbeat_ms` 1×/s (well under the 10 Hz ADR ceiling); shm-only so dante/null paths untouched. | Stale detection latency ≤ 1 s (acceptable: the warning is rate-limited to 1/30 s anyway). |
| (b) 10 Hz (every other `sleep_for(50ms)` iteration). | Matches the ADR "10 Hz" line literally. | No benefit over 1 Hz (source rate-limits dominate) and 10× the `system_clock`/header reads for nothing. The D-5 watchdog already chose 1 Hz for the same reason (`PlayerStaleWatchdog.h:22-25`). |
| (c) Tie to the `sleep_for` (20 Hz). | Simplest. | Far over-polls; same con as (b) amplified. |

### Recommended: **(a) shm-gated 1 Hz tick, placed immediately after the D-5 watchdog block.**

**Rationale.** Symmetry with the proven D-5 watchdog: same `if`-block region, same ~1 Hz `last_*_check` gate pattern, reusing the `now` (steady) and `now_unix_ms` (wall-clock) values already computed at `:583-591`. 1 Hz is correct because every `poll_diagnostics` warning is rate-limited at the source (30 s / 5 s / once), so a faster tick yields no extra warnings — exactly the reasoning `PlayerStaleWatchdog.h:22-25` documents for the 1 Hz watchdog. The block:

```
#if defined(__linux__) || defined(__APPLE__)
    if (input_is_shm && (now - last_shm_diag_check >= std::chrono::seconds(1))) {
        last_shm_diag_check = now;
        const std::uint64_t now_steady_ns = static_cast<std::uint64_t>(
            std::chrono::duration_cast<std::chrono::nanoseconds>(
                now.time_since_epoch()).count());
        shm_backend->poll_diagnostics(static_cast<std::uint64_t>(now_unix_ms),
                                      now_steady_ns);                      // D2
        shm_telemetry.tick(*shm_backend, engine.oscBackend(),
                           static_cast<std::uint64_t>(now_unix_ms));        // D1+D3
    }
#endif
```
where `shm_telemetry` is a `spe::bin::ShmTelemetryEmitter` declared once before the loop (guarded), and `last_shm_diag_check` is a `steady_clock::time_point{}` (default-constructed → first iteration fires immediately, seeding the prev-caches per D1/D3). The `now_unix_ms` is hoisted out of the D-5 watchdog block's inner scope (currently scoped at `:586-594`) so both ticks share it — a 1-line refactor (compute `now_unix_ms` once at the top of the loop body, pass to both `servicePlayerStaleWatchdog` and the shm tick).

## §0.5 — Deliberate-mode escalation rationale

PR4 is DELIBERATE for four reasons. (1) **Cross-process clock-domain contract (D2):** a wrong `now_ms` clock silently makes stale-detection always-fire or never-fire — a correctness bug invisible to a same-process test (the test injects the clock) that only surfaces in the PR6 cross-process soak; the contract must be pinned now and recorded as a PR5 obligation. (2) **Edge-on-a-live-RT-counter (D1/MJ-1):** `underrun_warnings_` is incremented on the RT thread with NO source rate-limit while the emitter diffs + 1/s-gates it on the control thread; a missed-delta, double-emit, or broken 1/s latch is intermittent and load-dependent. (3) **No-peer edge loss (CR-2):** the realistic case — a producer that dies/never-starts before a UI/soak client connects — would silently drop the very warnings PR6 is contracted to assert, unless the edge is held across the no-peer window; a fire-and-forget emit (the D-5 default) passes every same-process test yet fails the soak. (4) **Additive wire extension that downstream PRs assert across processes (P5):** the `/sys/warning shm_*` codes + `/sys/state` keys PR4 emits become the contract PR6's soak asserts and PR5's producer must make fire — a wrong code string or detail shape is a wire break discovered late. The pre-mortem + expanded test plan exist to catch these.

## §Pre-mortem — failure scenarios

- **PM1 — Double-emit / per-tick spam (D1/P2 over-fires).** If the emitter emits on every tick instead of on the counter EDGE — e.g. it forgets to update its prev-cache, or seeds the cache to 0 and treats a pre-existing non-zero count as a fresh edge every tick — a stale producer would spew `/sys/warning shm_producer_stale` at 1 Hz instead of once per 30 s, defeating the `poll_diagnostics` rate-limit and flooding the wire. **Detection:** ctest case "no-double-emit": drive the emitter with the SAME counter values across ≥ 3 ticks (no `poll_diagnostics` change) and assert EXACTLY zero new `/sys/warning` packets after the seed tick; a second case advances the stale counter once and asserts exactly ONE packet across 3 ticks. **Mitigation:** the emitter is a pure delta over the prev-cache; the FIRST tick seeds without emitting (D1, seed-silent row of the CR-2 table). **Cache-advance rule (reconciled with CR-2):** a code's prev-cache advances to the current count ON A SUCCESSFUL `sendReply` (true) — NOT unconditionally every tick. When `sendReply` returns true the cache catches up (so no re-emit); when it returns false (no peer) the cache is HELD behind so the edge is re-attempted (PM6). On a tick with NO count change AND a peer present, the diff is zero → no emit and the cache is already current. This single rule subsumes both "don't double-emit on a delivered edge" (PM1) and "don't lose an edge on no-peer" (PM6).

- **PM2 — Wrong clock domain → stale never/always fires (D2).** If the tick feeds `now_steady_ms` as `now_ms`, the `now_ms - producer_heartbeat_ms` difference is off by the unix epoch (~1.7e12 ms) and stale ALWAYS fires; if it feeds the right clock but PR5 writes steady-ms into `producer_heartbeat_ms`, stale never/always fires cross-process. **Detection:** the emitter ctest prefills `producer_heartbeat_ms` with a wall-clock value `now_unix_ms - 5000` (5 s stale) and `now_unix_ms - 10` (fresh), drives `poll_diagnostics(now_unix_ms, …)`, and asserts the stale counter edges exactly in the stale case and not the fresh case — proving the tick passes `now_unix_ms`. The PR5 producer-side half is PR4-Q1 (soak-verified in PR6). **Mitigation:** D2 pins `now_ms = now_unix_ms`; the tick reuses the loop's existing `now_unix_ms`; the contract is documented at the `poll_diagnostics` call site + recorded as the PR5 obligation.

- **PM3 — `/sys/state` latch never clears / clears prematurely (D3).** If the retry-on-no-peer latch clears regardless of `sendReply`'s return, a state change emitted while no peer is connected is LOST (the soak/operator never learns the producer drained/closed). Conversely, if it re-arms every tick it spams. **Detection:** ctest "state-latch retry": arm a state change with NO peer captured (assert `sendReply` returns false, latch stays armed), then attach a loopback peer and assert the NEXT tick delivers the packet and clears the latch (mirrors `test_osc_outbound_reply_smoke.cpp`'s peer-capture-then-reply flow + the `fallback_mode` latch semantics). **Mitigation:** clear the per-field latch ONLY when `sendReply` returns true (the exact `vst3/SpatialEngineProcessor.cpp:1362-1367` pattern); arm on value-change, not per-tick.

- **PM4 — Underrun edge torn/missed under RT increment (D1).** `underrun_warnings_` is `++`'d on the RT thread (`:417`) while the control-thread emitter reads it. A torn read (impossible on a 64-bit aligned `unsigned long long` on x86-64/ARM64 — the same word the existing `driver->xrunCount()` reads at the teardown printf `:636-638`) or a delta lost because two RT increments straddle one slow control tick. **Detection:** the emitter coalesces — it emits ONE edge per tick where the count increased (1/s-gated per MJ-1), with the detail = the cumulative total, so a multi-increment tick loses no information (the total is monotone). Ctest: advance the underrun count by 3 between two ticks; assert exactly ONE `/sys/warning shm_underrun` with detail = the current total. **Mitigation:** read each counter ONCE per tick into a local, diff against prev, emit on `> prev`, carry the cumulative total in the detail string (D1 N-at-once clause); benign-by-arch read documented (fold-in 7c). No new RT-path code (P1).

- **PM5 — Tick regresses the non-shm path / breaks JUCE-OFF build (D4/P3).** If the diagnostics tick is not `if (input_is_shm)`-gated, a dante/null run dereferences a null `shm_backend` → crash; if the `#include "bin/ShmTelemetryEmitter.h"` or the tick isn't under the `#if defined(__linux__)||defined(__APPLE__)` guard, a non-POSIX build fails to compile. **Detection:** AC-7 runs `--input-backend dante`/`--backend null` (no shm) and asserts no crash + identical behavior; the JUCE-OFF build (CI) compiles the guarded tick. **Mitigation:** the tick + include are inside the existing `shm_backend` guard and `if (input_is_shm)`-gated (D4), exactly mirroring the PR3 `shm_backend` usage at `:524/541`.

- **PM6 — `/sys/warning` edge LOST before a peer connects (CR-2, the realistic failure).** The retry-on-no-peer latch was historically applied ONLY to `/sys/state ,s`; the `,iis` warning path at `SpatialEngine.cpp:1074` (D-5) is fire-and-forget (ignores the `sendReply` return). A warning edge that fires before any OSC peer is captured — the COMMON case: a producer that never starts or dies early, before a UI/soak client connects (the loop's one-shots already wait `emit_after_ms`=1000 ms for peer capture, `spatial_engine_core.cpp:567-569`) — would be dropped FOREVER: `shm_attached_no_data`'s `poll_diagnostics` once-latch (`:505`) never re-fires, and stale won't re-fire for 30 s. PR6's soak (`test_phase_c_shm_loopback.py:207,:248`) is contracted to assert these cross-process — impossible if the edge is unrecoverable. **Detection:** AC-5b — advance a warning counter with NO peer, assert the edge is HELD (prev-cache not advanced, `sendReply` false, no packet); capture a peer; assert the next tick delivers it exactly once. **Mitigation:** CR-2 option (a) — the emitter does NOT advance a code's prev-cache until that code's packet is `sendReply`-confirmed true; the held edge is re-attempted each tick (the same latch the `/sys/state` path uses, now applied to `,iis`). This closes the fire-and-forget gap so PR6's cross-process assertion is satisfiable.

- **PM7 — Fast `producer_state` transition under-sampled by the 1 Hz tick (MJ-2).** A `Draining(2) → Closed(3)` transition shorter than the 1 Hz tick period is coalesced (only the latest `producer_state()` per tick is read), so `shm_producer_state=2` may never emit, though ADR §6 (`:225-226`) treats Draining and Closed as distinct operator states. **Detection:** documented bound in D3; PR4 does NOT test sub-tick transitions (out of scope — no producer→consumer state-change signal on the wire). **Mitigation:** PR4-Q6 — either PR5's producer holds each non-terminal state ≥ one tick period (producer obligation, like PR4-Q1) OR PR6 asserts only the terminal `Closed` state cross-process (intermediate-state observation = best-effort). Accepted for PR4; the 1 Hz coalescing bound is stated explicitly so it cannot silently surprise a future reader.
