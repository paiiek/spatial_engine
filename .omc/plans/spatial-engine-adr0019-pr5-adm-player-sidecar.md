## §0 — RALPLAN-DR Summary

### Principles (P)
- **P1 — Byte-for-byte wire fidelity to the FROZEN `RingHeader.h`.** The producer is a SECOND implementation of a contract the C++ consumer already ships and PR3's cases 9/10 already exercise. Every offset, every type, little-endian, `#pragma pack(1)` (no padding), the 4096-byte header size, the planar `4096 + c*capacity_frames*4` channel layout, and `capacity_frames` power-of-two are NOT PR5's to choose — they are READ from `RingHeader.h` and asserted byte-exact (AC-1). A single wrong offset is a silent cross-process corruption invisible to a Python-only test unless the test pins the exact bytes (which AC-1 does).
- **P2 — The audio loop is sacred (Python side, the producer's RT-equivalent), but the producer is explicitly NOT hard-RT.** `IpcRingSink.write()` is called from `play_adm_wav`'s playback loop at the 256/512-frame cadence. It must NOT block the loop (ADR §4.2: "preferable to blocking the Python audio loop"): the xrun policy is **drop-newest (drop the incoming block) + `xrun_count += 1`**, never a spin-wait on a full ring. No allocation churn in the steady state beyond the unavoidable numpy slices (D4 minimizes copies: one de-interleave per channel + the wrap-split memcpy). **Honest scope (iter-2 fix, Critic MAJOR):** unlike the C++ side, the publish path makes exactly ONE deliberate scheduler syscall — `os.sched_yield()` (D-publish) — which is acceptable because the Python producer is a non-hard-RT player loop (ADR §1.3), not the engine's RT audio callback. The earlier "no syscalls in the publish" wording was self-contradictory and is corrected: the publish path is allocation-free and lock-free, and its single `sched_yield()` is a best-effort scheduler nudge, NOT a memory fence (see P3).
- **P3 — Publish ordering: PR5 owns the program-ORDER, PR6 owns the memory-ORDERING proof (iter-2 reframe, Critic CRITICAL C1 + MAJOR).** The consumer load-acquires `write_idx` and then reads exactly the frames below it. The producer MUST emit the stores in strict program order: (1) copy the channel samples, (2) stamp `producer_meta_block_pts_ns`, (3) THEN store the advanced `write_idx` LAST. **What PR5 can and cannot prove:** Python has no explicit store-release. On x86-64 the hardware memory model (TSO) makes prior stores visible-before a later store, so program order ⇒ the consumer's `write_idx`-acquire pairs correctly. On ARM64 this is **NOT guaranteed** by `os.sched_yield()` — `sched_yield(2)` is a scheduler hint with NO documented memory-ordering semantics (it emits no `dmb`/release barrier); it is a best-effort nudge, NOT a fence. So at v0.8 the SPSC release contract is **x86-TSO-correct and ARM-UNPROVEN**, and the cross-language / cross-process memory-ordering proof is **OWNED BY PR6's concurrent soak** (which actually races a C++ acquire-load against the Python release-store under load on real hardware). **AC-9 in PR5 is therefore a PROGRAM-ORDER / logic structural check only** — it asserts the producer issues the stores in the right source order and that every frame below a published `write_idx` is the fully-written expected value *in a serialized read* — it does NOT (and under the GIL cannot) prove the memory-ordering pairing. Getting the program order wrong (issuing the `write_idx` store before the sample copy) is the bug AC-9 DOES catch; the weak-memory reordering risk is PR6's. This honesty is the iter-2 correction of the iter-1 over-claim that AC-9 "proves the SPSC release contract."
- **P4 — Producer owns the shm lifecycle; the consumer never unlinks; and `close()` MUST release every buffer view before `SharedMemory.close()` (iter-2 fix, Critic CRITICAL C2).** Verified `SharedMemoryRegion.cpp:155-156` ("does NOT call shm_unlink — the producer owns the lifecycle"). So `IpcRingSink` creates (`create=True`), zero-inits `_reserved` (→ consumer-lock word starts unlocked, ADR §2.3 footnote — PR3 consumer CAS-locks it, PR5 producer NEVER touches it after the initial zero), and on `close()` does `SharedMemory.close()` + `.unlink()`. **CRITICAL ordering (reproduced on this host, Python 3.12.3):** the per-channel `np.frombuffer(shm.buf, …)` ring views (and any header `memoryview`) are live *exporters* of `shm.buf`; calling `SharedMemory.close()` while they exist raises `BufferError: cannot close exported pointers exist`. So `close()` must FIRST drop every retained view (set the per-channel view attributes + any header memoryview to `None`), then `gc.collect()` to force release of the buffer exporters, THEN `SharedMemory.close()` + `.unlink()`. (Verified: a transient `struct.pack_into(shm.buf, …)` does NOT linger — only *retained* numpy/memoryview references block close.) Without this, `__exit__`→`close()` throws and LEAKS `/dev/shm/<name>` — the exact stale-object leak P4 claims to prevent. The context-manager guarantees the (now correct) teardown on every exit path. **AC-13 pins this (construct→write→close succeeds with no `BufferError`, `/dev/shm/<name>` gone after).**
- **P5 — adm_player's existing OSC / sync / playback contract is unchanged.** `--sink ipc://` ONLY swaps the audio sink (`sd.OutputStream` → `IpcRingSink`). The ADM-OSC emitter (`/adm/obj/N/aed`, 9000) and Phase B sync (`/sys/handshake`, `/hb/ping`, `/transport/play`, `osc_sync.py`) paths are byte-identical — verified both `osc_emit.py` and `osc_sync.py` are sink-agnostic. The default (no `--sink ipc://`) playback path is unmodified. The full existing adm_player pytest stays green (AC-10).
- **P6 — Discharge PR4's producer obligations exactly as PR4 contracted them.** PR4-Q1: `producer_heartbeat_ms` = unix-epoch ms (`int(time.time()*1000)`) — PR4's stale detector compares `now_unix_ms - producer_heartbeat_ms`; a steady-ms heartbeat breaks it cross-process (D2). PR4-Q6: hold each non-terminal `producer_state` ≥ one engine-tick period (1 s) so the engine's 1 Hz telemetry tick observes it, OR document terminal-`Closed`-only observability (D2). PR5 makes these producer-side facts true and testable in-process; PR6 verifies them cross-process.
- **P7 — Cross-repo commit-boundary is a first-class risk (the real footgun); the dreamscape tree is VOLATILE, not clean (iter-2 fix, Critic MAJOR + Architect Condition 1).** The PR5 PRODUCER CODE lives in the **`/home/seung/mmhoa/adm_player/dreamscape`** git repo (its OWN repo — verified `rev-parse --show-toplevel`; NOT a submodule). ## Decision 1 — Round-trip test strategy (hardware-free)

The PR5 pytest must prove the producer writes a correct ring on a 1-second buffer (ADR §8 step 5) WITHOUT a real audio device and WITHOUT a fragile cross-repo build dependency on the C++ engine.

### Viable Options

| Option | Mechanism | Pros | Cons |
|---|---|---|---|
| **(a) Pure-Python round-trip.** The test writes via `IpcRingSink` then reads the ring back with a small in-test Python reader that mirrors the C++ consumer's masked-index read (`offset = 4096 + c*capacity*4 + (i & (capacity-1))*4`, ADR §2.4) + the header byte layout pinned against `RingHeader.h`. | Self-contained in the adm_player repo (P7); no C++ build/toolchain dependency; fast; deterministic; pins the EXACT header bytes + planar layout + indices (AC-1..AC-9). ## Decision 2 — `IpcRingSink` lifecycle + state machine + heartbeat (discharges PR4-Q1 + PR4-Q6)

The producer drives `producer_state` (Idle=0/Streaming=1/Draining=2/Closed=3, `RingHeader.h:31-36`) and `producer_heartbeat_ms`. PR4's consumer-side telemetry (`/sys/state shm_producer_state`, `shm_producer_alive` from the 100 ms stale threshold, `/sys/warning shm_producer_stale`) is contingent on the producer writing the RIGHT clock and holding states long enough to be observed at 1 Hz.

**Verified frozen facts:** the engine stale threshold is **100 ms** (`producer_heartbeat_ms` stale > 100 ms = producer died, ADR §2.3 line 81; `SharedRingBackend.h:77` `kStaleThresholdMs`); the engine's diagnostics tick is **1 Hz** (PR4 D4, `spatial_engine_core.cpp:613` `last_shm_diag_check >= std::chrono::seconds(1)`); ADR §6 line 166 notes the engine's `HeartbeatPublisher` runs at 10 Hz on the control thread.

### Viable Options

| Axis | Option | Pros | Cons |
|---|---|---|---|
| **Heartbeat clock (PR4-Q1)** | **(h1) unix-epoch ms `int(time.time()*1000)`** | EXACTLY what PR4's `now_ms - producer_heartbeat_ms` compares against (ADR §2.3 line 81 "Wall-clock ms"); cross-process stale detection correct. | Wall-clock non-monotonic (NTP step) — same bounded exposure PR4 D2 already accepted; rare, self-corrects. |
| | (h2) `time.monotonic()*1000` (steady) | Monotonic. | WRONG: off by the unix epoch (~1.7e12 ms) vs the consumer's wall-clock `now_ms` → stale ALWAYS fires cross-process. Rejected (this is the exact PR4-Q1 trap). |
| **Heartbeat cadence** | **(c1) beat on every `write()` AND from a 10 Hz background timer thread** | The 100 ms threshold needs a beat faster than 100 ms; per-block beats cover the streaming case (256/48000 ≈ 5.3 ms ≪ 100 ms), and the 10 Hz timer covers gaps (pause / between-track / slow disk read) so the producer isn't falsely flagged stale while alive. Matches the engine's 10 Hz `HeartbeatPublisher` cadence (ADR §6:166). | One background `threading.Thread` (daemon) + a stop event; modest. |
| | (c2) beat only on `write()` | Simplest. | A legitimate gap > 100 ms (pause, between-block disk stall) falsely trips `shm_producer_stale`/`shm_producer_alive=0`. Rejected — the 100 ms threshold is tight relative to non-RT Python scheduling jitter. |
| **State-transition hold (PR4-Q6)** | **(s1) hold each non-terminal state ≥ 1 engine-tick period (1 s) before the next transition; on `close()`: Streaming→Draining, dwell ≥ 1 s, →Closed** | The engine's 1 Hz tick observes every transition (Draining distinctly from Closed); discharges PR4-Q6 option (a) — the producer obligation. | `close()` blocks ~1 s on the drain dwell (acceptable on teardown; or make the dwell skippable via a flag, see rationale). |
| | (s2) Idle→Streaming→Closed, no dwell; document terminal-`Closed`-only observability | No teardown delay. | The 1 Hz engine tick may coalesce a fast Draining→Closed (PR4-Q6/MJ-2 / PM7); `Draining` may never be observed. PR4-Q6 option (b) — best-effort intermediate states. |

### Recommended: **(h1) unix-epoch ms; (c1) per-write + 10 Hz background heartbeat; (s1) Idle→Streaming on first write, and on close() Streaming→Draining (dwell ≥ 1 engine tick) →Closed — discharging PR4-Q6 option (a).**

**Rationale.** **h1** is forced by ADR §2.3 line 81 + PR4-Q1 (BLOCKING-FOR-PR5): the consumer's `now_ms` is wall-clock, so the producer heartbeat must be wall-clock unix-epoch ms; h2 is the documented always-fires trap. **c1**: the 100 ms stale threshold (`SharedRingBackend.h:77`) is tight; per-write beats cover active streaming (a 256-frame block at 48 kHz is ~5.3 ms apart, far under 100 ms) but a pause/seek/disk-stall gap > 100 ms would falsely flag a live producer — so a daemon 10 Hz timer thread beats independently while the sink is open (mirroring the engine's own 10 Hz `HeartbeatPublisher`, ADR §6:166). The timer stops at `close()`. **s1**: PR4-Q6 offers two contracts; PR5 picks the stronger producer obligation (option a) so the soak can assert `Draining` distinctly from `Closed` — `Idle(0)` at construction, `Streaming(1)` stamped on the first `write()`, and `close()` does `Streaming→Draining(2)`, dwells one engine-tick period (1 s, so the engine's 1 Hz tick can sample `Draining` — PR4 D3/PM7), then `Closed(3)` + final heartbeat + unlink. The AC-7b drain-dwell test uses a deterministic handshake (a `threading.Event`/`Condition` the producer sets on entering `Draining`, or an injected clock), NOT wall-clock `time.sleep` polling of `producer_state` from another thread, which is flaky under scheduler jitter (Critic test-adequacy). **Non-blocking RT-hygiene refinement (Architect synthesis):** the daemon timer MAY be a "beat-if-idle" timer (beat only when `write()` has not beaten within ~50 ms) so it never fires during active streaming (256 frames @ 48k ≈ 5.3 ms cadence) — eliminating GIL contention on the hot path while still covering pause/seek/disk-stall gaps; this preserves c2's "no extra thread activity during streaming" with c1's "no false-stale during gaps". Optional for PR5; the unconditional 10 Hz daemon is the acceptable baseline.

## Decision 4 — de-interleave + write-path layout (planar ring from interleaved blocks)

The existing playback loop produces an INTERLEAVED 2D `np.ndarray` shape `(frames, channels)`, C-contiguous, float32 (`playback.py:227` `f.read(..., always_2d=True)`, `:231` `np.asarray(..., order="C")`, handed to `stream.write` at `:257`). The shm ring is **PLANAR** (per-channel contiguous, `RingHeader.h:148-153` `channel_byte_offset(c) = 4096 + c*capacity_frames*4`; ADR §2.2 "Channels are planar, not interleaved"). So `IpcRingSink.write()` must de-interleave on write, and copy into a power-of-two ring with wrap-around.

### Viable Options

| Axis | Option | Pros | Cons |
|---|---|---|---|
| **De-interleave** | **(d1) `block[:, c]` per channel → contiguous copy into the channel's ring slice** | Trivial, correct, readable; numpy column slice is a strided view, the ring write forces the contiguous copy anyway. One pass per channel. | A strided read per channel (cache-unfriendly for many channels, but the block is tiny — 256×N floats). |
| | (d2) `block.T.copy()` once (transpose to `(channels, frames)` planar) then per-channel contiguous slices | One transpose, then contiguous per-channel. | Allocates a transposed copy each block; for the wrap-split it still needs per-channel two-part copies. Marginal; (d1) is simpler and avoids the full-block temp. |
| **Wrap-around** | **(w1) split each channel write into ≤2 contiguous copies at the `i & (capacity-1)` boundary** (head fills to end-of-ring, tail wraps to ring start) | Exactly the ADR §2.4 masked-index semantics; the C++ consumer reads the same way; correct for any block straddling the ring end. | Two `np.ndarray`/`memoryview` slice-assigns when straddling; one when not. |
| | (w2) require `capacity_frames % block_size == 0` and write whole blocks only | Simpler index math. | Still wraps at capacity; doesn't remove the split, only aligns it; over-constrains capacity. Rejected — w1 handles the general case. |

### Recommended: **(d1) per-channel strided slice → contiguous ring copy, (w1) ≤2-part wrap-split per channel at the `i & (capacity-1)` boundary.**

**Rationale.** The ring is planar and the consumer indexes `(i & (capacity-1))` per channel (ADR §2.4, mirrored in `channel_byte_offset`), so the producer must write each channel's `block_size` frames into `[write_idx & (capacity-1) ...]` with a wrap-split when the run crosses the ring end — w1 is the faithful mirror. d1 (per-channel `block[:, c]`) is the simplest correct de-interleave; the block is small (256–512 frames × N channels) so the strided read cost is negligible and avoids the full-block transpose temp of d2 (keeping per-block allocation minimal, P2). The ring memory is viewed as `np.ndarray`s over the `SharedMemory.buf` with EXPLICIT little-endian dtype (`np.dtype("<f4")`, NOT native `np.float32` — the wire is little-endian per `RingHeader.h`, so this is correct on a hypothetical big-endian host too) created once at construction as per-channel offset slices, so each write is a vectorized slice-assign, not a Python loop. **These persistent views are buffer exporters of `shm.buf` and MUST be dropped + `gc.collect()`-ed before `SharedMemory.close()` (P4/AC-13, Critic C2).** **Free-space / xrun check FIRST** (ADR §4.2): `free = capacity_frames - (write_idx - read_idx)` (read `read_idx` load-acquire); if `free < block_size`, **drop the INCOMING block (drop-newest) + `xrun_count += 1`** and DO NOT advance `write_idx` — never block the loop (P2). **iter-2 naming fix (Architect Condition 3 / Critic Minor 1):** the policy is unambiguously **drop-newest (discard the incoming block; `write_idx` is NOT advanced, so nothing already-published is overwritten)** — the iter-1 "drop-oldest"/"overwrite-oldest" wording was wrong and is reconciled here; ADR §4.2's "overwrite the oldest" phrasing is realized as drop-incoming (the producer never rewinds `write_idx`, hence cannot overwrite resident frames — overwriting would race the consumer's read). AC-4 (planar correctness) + AC-5 (wrap) + AC-6 (xrun drop, deterministic assertion: `write_idx` unchanged + `xrun_count += 1`) pin this.

## Decision 5 — `capacity_frames` pow2 padding + `block_size` reconciliation

ADR §2.3/§2.4: `capacity_frames` MUST be power-of-two (the C++ consumer **rejects non-pow2 at attach** — PR2-Q4: "Non-pow2 is rejected at `start()`"). ADR §2.4: "We pad to the next power of two if the user requests `--ring-frames 6000` — the engine logs 'padded to 8192' once at start." `block_size` is a wire field (`RingHeader.h:83` @ `0x0014`) the consumer validates as a divisor of the engine's block (PR2-Q7 `BlockConfigMismatch`). The existing CLI default is `--block-frames 512` (`__main__.py:40`); the ADR §4.1 example uses `--block-size 256`.

### Viable Options

| Axis | Option | Pros | Cons |
|---|---|---|---|
| **pow2 padding** | **(p1) pad `ring_frames` up to next pow2 in `IpcRingSink.__init__`, log once** | The producer OWNS the pad (ADR §2.4 is a "producer-side obligation" per PR2-Q4); writes the padded value into `capacity_frames` so the consumer attaches cleanly. | None material — it's the spec. |
| | (p2) require caller to pass a pow2 | Stricter. | Pushes the ADR §2.4 producer obligation onto the CLI/caller; worse UX. Rejected. |
| **block_size source** | **(b1) new `--block-size` (default 256, ADR §4.1) used ONLY for the shm wire `block_size`; keep `--block-frames` (default 512) for the audio read granularity; in shm mode the read block = the shm `block_size`** | Honors the ADR §4.1 example; `block_size` is the producer's write granularity (ADR §2.3:75 "Typical 64/128/256"). | Two block knobs; reconcile so the read loop feeds the sink in `block_size` chunks. |
| | (b2) reuse `--block-frames` (512) as the shm `block_size` | One knob. | Diverges from the ADR §4.1 example (256); 512 still valid (≤ MAX_BLOCK 512, `RingHeader.h` consumer gate). Acceptable fallback. |

### Recommended: **(p1) producer pads `ring_frames`→next pow2 (default 8192) + logs once; (b1) add `--block-size` (default 256) as the shm wire `block_size`, the playback read block in shm mode = `block_size`.**

**Rationale.** **p1** is the ADR §2.4 producer obligation (PR2-Q4 confirmed the consumer never rounds — it rejects non-pow2; padding is producer-side). `__init__` computes `capacity = next_pow2(ring_frames)`, writes it to `capacity_frames`, and logs `"padded to <capacity>"` once when `ring_frames` was not already pow2 (matching the ADR §2.4 wording). Default `--ring-frames 8192` (already pow2, ~170 ms @ 48 k, ADR §2.3:77) — no pad in the common case. **b1**: the ADR §4.1 example is `--block-size 256`, and `block_size` is semantically the producer's *write granularity* (ADR §2.3:75) distinct from `--block-frames` (the player's existing audio read/OSC block, default 512). In shm mode the read loop reads `block_size`-frame chunks and feeds each to `sink.write()` so the wire `block_size` matches what's actually written; `--block-frames` keeps its meaning for the device path. The consumer validates `block_size` divides the engine block (PR2-Q7) — 256 divides a 256/512 engine block cleanly. **iter-2 operator contract (Architect suggestion 4 / Critic "What's Missing"):** the verified gate direction is `engine_block_size % header.block_size == 0` (`SharedRingBackend.cpp:141-146` → `BlockConfigMismatch`), i.e. the PRODUCER's `block_size` must DIVIDE the engine's callback block. Concretely: `--block-size 256` with engine block 256 or 512 is OK (256%256=0, 512%256=0); `--block-size 512` against engine block 256 would FAIL (256%512≠0) and the engine refuses to attach. The operator MUST run the engine with a block that is a multiple of `--block-size`. **Recorded as PR5-Q4 (ratify `--block-size` default 256 as the shm wire granularity vs reusing `--block-frames` 512; and the `engine_block % block_size == 0` divisor contract the operator must respect).**

## §Pre-mortem — failure scenarios

- **PM1 — A wrong header offset / type / endianness silently corrupts the ring (P1/D1).** If `ipc_sink.py`'s offset table drifts from `RingHeader.h` (e.g. `producer_heartbeat_ms` at the wrong offset, a `u32` written where a `u64` is expected, or a big-endian `struct` format), the C++ consumer reads garbage — invisible to a Python test that uses the SAME wrong constants for read-back. **Detection:** AC-1 asserts the literal header bytes/offsets **transcribed directly from `RingHeader.h`** (magic == `0x53504543484D4E47` little-endian @ `0x0000`, version==1 @ `0x0008`, header_size==4096 @ `0x000C`, and **ALL 15 field offsets** — iter-2 fix), NOT the producer's own constants — so a producer/reader shared bug still fails the literal-byte assertion; PR6's soak is the cross-language backstop. **iter-2 gap closed (Architect Condition 2 / Critic MAJOR 1):** iter-1's AC-1 omitted the THREE unaligned hot-path atomics the producer actively writes — `producer_heartbeat_ms`@0x0034, `xrun_count`@0x003C, `producer_meta_block_pts_ns`@0x0044 — which are exactly the highest-risk fields (4-mod-8 unaligned, most prone to a packing/offset slip) and the named PM1 scenario; a wrong heartbeat offset would pass iter-1 AC-1 entirely (AC-7a only checks the heartbeat VALUE read back via the producer's own — possibly wrong — constant). AC-1 now pins these three by literal offset (assert they read 0 at construction, and the just-stamped heartbeat reads back at exactly 0x0034), plus the index `<Q` u64 WIDTHS (`write_idx`@0x0020, `read_idx`@0x0028, `seq`@0x0050 are u64, not u32 — a `<I` slip passes a 1 s test but corrupts after the 2^32 frame boundary). **Mitigation:** a single `_HEADER` offset module in `ipc_sink.py` with an inline `# RingHeader.h:<line>` comment per field; all `struct.pack_into` use explicit little-endian formats (`"<Q"`, `"<I"`, `"<f"`); AC-1 pins all 15 offsets + widths.

- **PM2 — Wrong publish PROGRAM-order → torn frames; AND weak-memory reordering on ARM → torn frames (P3/D-publish; split in iter-2 per Critic C1).** Two distinct failure modes: (2a) `write()` issues the `write_idx` store BEFORE the channel copies / pts stamp in SOURCE order — a logic bug; (2b) on ARM64 the hardware reorders an otherwise-correctly-ordered store so the consumer's acquire-load sees `write_idx` advanced before the sample stores land — a weak-memory bug `os.sched_yield()` does NOT prevent (it is not a fence). **Detection of (2a) [PR5/AC-9]:** AC-9 instruments the store sequence (a recording wrapper) to assert SOURCE order is copy→pts→write_idx, and a serialized read asserts every frame below `write_idx` is the fully-written ramp + `producer_meta_block_pts_ns` is non-zero/monotonic across published blocks. **(2b) is NOT detectable in-process** — a GIL-serialized same-process reader always sees completed stores regardless of memory order (empirically confirmed: a deliberately reordered publish still reads complete in-process). **Detection of (2b) [PR6 ONLY]:** PR6's concurrent cross-process C++↔Python soak races a real C++ acquire-load against the Python release-store on real hardware. **Mitigation:** strict source order in `write()`: copy channels → stamp pts → `os.sched_yield()` (best-effort scheduler nudge, NOT a fence) → store `write_idx` LAST; x86-64 TSO makes this correct, ARM64 deferred to PR6. The iter-1 claim that AC-9 "proves the SPSC release contract" is corrected — AC-9 proves (2a), PR6 proves (2b).

- **PM3 — Heartbeat in the wrong clock → cross-process stale always/never fires (P6/D2/PR4-Q1).** If `producer_heartbeat_ms` is written as `time.monotonic()*1000` instead of `int(time.time()*1000)`, the consumer's wall-clock `now_ms - producer_heartbeat_ms` is off by the unix epoch → `shm_producer_stale`/`shm_producer_alive=0` always fires. **Detection:** AC-7a asserts `producer_heartbeat_ms` is within a few seconds of `time.time()*1000` (i.e. unix-epoch, not steady — a steady value would be ~1.7e12 ms smaller). **Mitigation:** D2 h1 pins `int(time.time()*1000)`; the per-write + 10 Hz timer both use it; documented as the PR4-Q1 discharge.

- **PM4 — The xrun policy blocks the Python audio loop (P2/D4).** If `write()` spin-waits or sleeps when the ring is full instead of dropping, a slow consumer stalls the player's playback loop (audio glitches, OSC emit stalls). **Detection:** AC-6 fills the ring (no consumer advancing `read_idx`), then asserts — DETERMINISTICALLY (iter-2, Critic) — that after the dropping `write()` the `write_idx` is UNCHANGED and `xrun_count` incremented by exactly 1 per dropped block; the "returns promptly / no block" wall-clock timing is an ADVISORY secondary check only (not the pass/fail gate, which would be flaky under CI load). **Mitigation:** D4 — free-space check FIRST; `free < block_size` → drop the incoming block + `xrun_count += 1` + return; never block (ADR §4.2 drop-newest).

- **PM5 — `_reserved` / consumer-lock word not zero-inited → consumer attach refuses or mis-locks (P4/ADR §2.3 footnote).** If the producer leaves `_reserved` (incl. the consumer-lock word at `0x0058`) non-zero, the PR3 consumer's CAS `0→pid` fails (it sees a non-zero "already-locked" word) and refuses to attach (SPSC `multi_attach_rejected`, case 10). Or if the producer ever WRITES the lock word after init, it corrupts the consumer's lock. **Detection:** AC-8 asserts the entire `_reserved` span (`0x0058`..`0x1000`) is zero immediately after `__init__`, the lock word at `0x0058` is exactly 0, and that NO `IpcRingSink` method writes into `_reserved` after construction (assert it stays 0 across `write()`/heartbeat/`close()` except as the consumer would set it — which the producer never does). **Mitigation:** `__init__` does `shm.buf[0x0058:0x1000] = b"\x00" * (0x1000-0x0058)` (or relies on `SharedMemory` create zero-fill, then explicitly re-zeros to be safe); the producer NEVER touches `_reserved` again (P4).

- **PM6 — `capacity_frames` not power-of-two → consumer rejects at attach (D5/PR2-Q4).** If the producer writes a non-pow2 `capacity_frames` (e.g. passes `--ring-frames 6000` straight through), the C++ consumer's `start()` rejects it (`BackendError::DeviceOpenFailed`, PR2-Q4) and the engine exits nonzero — the run never starts. **Detection:** AC-2 passes `ring_frames=6000`, asserts `capacity_frames` in the header is `8192` (next pow2) and a one-time "padded to 8192" log fired; AC-2 also asserts a pow2 input (8192) writes 8192 with NO pad log. **Mitigation:** D5 p1 — `__init__` pads to next pow2 and logs once.

- **PM7 — A fast Draining→Closed transition is under-observed by the engine's 1 Hz tick (P6/D2/PR4-Q6).** If `close()` does Streaming→Closed with no Draining dwell, the engine's 1 Hz telemetry tick (PR4 D4) may never sample `Draining(2)` — `/sys/state shm_producer_state=2` never emits, though ADR §6:225-226 treats Draining/Closed as distinct. **Detection:** AC-7b — with `drain_dwell_s=1.0`, assert `close()` sets `producer_state=Draining(2)`, dwells ≥ 1 s, THEN `Closed(3)` (sample the state field during the dwell); with `drain_dwell_s=0.0`, assert the documented best-effort (terminal `Closed` only) per PR4-Q6 option (b). **Mitigation:** D2 s1 — default `drain_dwell_s=1.0` discharges PR4-Q6 option (a); the dwell is coupled to PR4's 1 Hz cadence (PR5-Q2).

- **PM8 — `git add -A` in the wrong (or volatile) repo sweeps unrelated dirty files (P7/D6).** If the executor runs `git add -A` (or `git commit -am`) from `/home/seung`, it stages 531 unrelated dirty files (claude_text2traj, etc.) into a single commit — a catastrophic, hard-to-unwind mess; or commits the producer code into the wrong (outer) repo that doesn't even track it. **iter-2 escalation:** the dreamscape repo is ALSO volatile — parallel sessions leave unrelated `adm_recorder` work dirty there (verified: 3 such files mid-review, since committed as `6b9534d`), so a `git add -A`/`commit -am` in DREAMSCAPE would also sweep in-flight unrelated work. The footgun is not confined to `/home/seung`. **Detection:** the execution checklist (D6) requires `git -C <repo-root> status --short` before EVERY add, scoped `git -C <repo-root> add <explicit path>` only (`git add -A` forbidden in BOTH repos), and a pre-commit `git -C <root> diff --cached --name-only` review confirming ONLY the intended PR5 paths are staged. **Mitigation:** D6 — code commits in `dreamscape` with scoped adds of exactly the 4 PR5 paths (re-checking the volatile tree first); the checklist spells out the exact `git -C` invocations; PM8 is the headline footgun.

- **PM9 — `IpcRingSink.close()` raises `BufferError: cannot close exported pointers exist` and LEAKS the shm object (P4/Critic CRITICAL C2).** The per-channel `np.frombuffer(shm.buf, …)` ring views are persistent buffer exporters; calling `SharedMemory.close()` while they are live raises `BufferError` — **reproduced on this host (Python 3.12.3)**. Every test that constructs+closes a sink (AC-3/AC-7b/AC-8/AC-10/AC-12) and the context-manager `__exit__` the CLI relies on would throw at teardown, leaving `/dev/shm/<name>` leaked — the exact stale-object leak P4 claims to prevent. **Detection:** new **AC-13** — construct → `write()` → `close()` returns with NO `BufferError`, and `/dev/shm/<name>` is gone afterward; also run under the context-manager (`with IpcRingSink(...)` then assert unlinked on exit). **Mitigation:** `close()` drops every retained view/memoryview (→ `None`) + `gc.collect()` BEFORE `SharedMemory.close()`+`.unlink()` (P4); idempotent double-close guard. (A transient `struct.pack_into(shm.buf, …)` is verified NOT to linger — only RETAINED references block close.)

- **PM10 — the `IpcRingSink` path runs device-specific logic and fails on a headless host (P5/D-seam/Critic ambiguity).** If the sink-seam refactor leaves `sd.query_devices`/`_output_device_id`/`resolve_playback_channels`/`max_output_channels` truncation (`playback.py:211-217`) in the SHARED path, the `--sink ipc://` mode queries an audio device that does not exist on a headless host — AC-11 fails and the "no hardware" goal is defeated; worse, channel-cap truncation would silently drop channels the ring expects. **Detection:** AC-11 runs HEADLESS (no audio device) and asserts the ipc path constructs `IpcRingSink` writing ALL `channels` with NO device query; a test on a no-device host (or with `sd` device calls stubbed to raise) must still pass. **Mitigation:** D-seam — the refactor extracts the OUTPUT-STAGING concern so the ipc path bypasses ALL device logic; only the device path keeps truncation; the existing `play_adm_wav` caller (`interactive.py:174`) signature is preserved.
