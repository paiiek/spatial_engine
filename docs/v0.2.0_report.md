## 1. Spatial Engine이 무엇인가 (Non-technical 요약)

### 1.1 한 문장 요약
**Spatial Engine은 라이브 공연장·실험실에 설치된 다수의 스피커에 "소리가 어느 위치에서 들리는지"를 실시간으로 제어해 주는 오디오 렌더링 엔진**입니다.

### 1.2 비유로 설명
- TV가 "픽셀의 색"을 그리는 도구라면, Spatial Engine은 **"공간 안의 소리 위치"**를 그리는 도구입니다.
- 콘솔(믹싱 보드)에 앉은 엔지니어가 "이 보컬을 무대 좌측 위쪽에 두고 싶다"고 지시하면, Spatial Engine이 그 의도를 받아 **수많은 스피커에 각각 얼마나 강하게/언제 소리를 보낼지** 계산해서 출력합니다.

### 1.3 실제 사용처
| 분야 | 시나리오 |
|---|---|
| 라이브 공연 | 가수 목소리가 무대 위 특정 지점에서 들리도록 — 관객 위치에 무관하게 일관된 공간감 |
| 영화·드라마 후반 작업 | 객체 기반 오디오(ADM, Dolby Atmos 호환) 믹싱 |
| 실험 음향학 | 가상 음원 위치를 정밀 제어해 청취자 인지 실험 |
| VR/메타버스 | 6-DoF 음향 시뮬레이션 |

### 1.4 왜 이런 도구가 필요한가
일반 "스테레오"는 좌우 두 채널뿐이라 무대 깊이·높이·후방을 표현하지 못합니다.
최근 라이브 베뉴는 **수십~수백 개 스피커**를 배치해 객체별 위치를 표현하지만,
콘솔→스피커 매핑은 수학적으로 복잡합니다.
Spatial Engine은 그 매핑 계산을 **실시간(≤5ms 목표)**에 수행합니다.

### 1.5 무엇이 새로운가 (경쟁 대비)
| 기능 | 상용 (L-ISA, Dolby) | Spatial Engine |
|---|---|---|
| ADM-OSC 수신 (산업 표준 OSC 프로토콜) | ✓ | ✓ (v0.2.0 신규) |
| VST3 플러그인 (DAW 내장 가능) | △ (별도 제품) | ✓ (v0.2.0 신규, JUCE-free) |
| 고차 앰비소닉 디코더 4종 | 부분 지원 | ✓ (MaxRE/AllRAD/EPAD/InPhase) |
| 오픈소스 (GPL v3) | ✗ | ✓ |
| 단일 워크스테이션 운용 | 추가 하드웨어 필요 | ✓ (Linux PC + Dante PCIe) |

## 2. 시스템 아키텍처 (한눈에)

```
┌─────────────────────────────────────────────────────────────────┐
│ 외부 컨트롤러                                                       │
│  • 믹싱 콘솔 (DiGiCo / Avid / Yamaha)                             │
│  • L-ISA Controller, Spat Revolution                            │
│  • 자체 PySide6 UI                                                │
└────────────────────────────┬────────────────────────────────────┘
                             │ OSC/UDP (포트 9100)
                             │ /adm/obj/N/{aed, gain, xyz, ...}
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│ Spatial Engine Core  (C++20, JUCE 7.0.12 옵션)                   │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │ OSC Decoder  →  CommandQueue (SPSC FIFO, lock-free)      │    │
│  └─────────────────────────┬────────────────────────────────┘    │
│                            ▼                                       │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │ Renderer  (실시간 오디오 스레드)                              │    │
│  │   • VBAP / WFS / DBAP / HOA (4종 디코더)                     │    │
│  │   • Per-object Source Width                                │    │
│  │   • FDN Reverb (16-line Hadamard) + IRConv (WAV/SOFA)     │    │
│  │   • ChannelLimiter, 시간정렬                                  │    │
│  └─────────────────────────┬────────────────────────────────┘    │
│                            ▼                                       │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │ Audio I/O Backend                                          │    │
│  │   • Dante (Digigram ALP-Dante PCIe) — 프로덕션                │    │
│  │   • Null (CI/개발용)                                          │    │
│  └─────────────────────────┬────────────────────────────────┘    │
└────────────────────────────┼────────────────────────────────────┘
                             ▼
                  ┌─────────────────────┐
                  │ 다채널 스피커 어레이      │
                  │ (8/16/32/64 채널)      │
                  └─────────────────────┘
```

### 2.1 모듈 책임
- **Core (C++)**: 실시간 DSP — 모든 RT-safe 경로 (할당 0, 락 0)
- **UI (PySide6)**: 시각화·드래그 조작 (RT 외부)
- **VST3 플러그인**: DAW 내장형 컨트롤러 (v0.2.0 신규)
- **Bridge (Python)**: vid2spatial 비전 추적·ADM-OSC 합성기

## 3. v0.2.0 주요 변경 사항 (5가지)

### 3.1 ADM-OSC v1.0 수신 호환 (Phase C3)
**무엇**: 산업 표준 ADM-OSC v1.0 OSC 패킷을 그대로 수신.
**왜 중요**: DiGiCo / Avid / Yamaha 콘솔, L-ISA Controller, Spat Revolution이
보내는 패킷을 별도 변환 없이 처리 가능. **국내 라이브 베뉴 첫 도입 시나리오의
핵심 게이트.**
**범위**:
- `/adm/obj/N/{azim, elev, dist, aed, gain, mute, xyz, active, width, name}` 10종
- 4종 신규 `CommandTag` 추가 (0x06–0x09)
- 88-row CSV 컴플라이언스 픽스처
- 60s × 64 obj × 1 kHz 실 UDP 부하 검증 (Soak harness)

### 3.2 HOA 디코더 4종 추가 (Phase M2-extended)
**무엇**: 고차 앰비소닉(HOA) 디코딩 알고리즘을 4종 확장.
**왜 중요**: 룸/스피커 레이아웃별 최적 알고리즘 선택 가능.
**알고리즘**:
- **MaxRE** — 최대 에너지 벡터 (Legendre-root 기반)
- **AllRAD** — t-design 쿼드러처 투영
- **EPAD** — Energy-Preserving 양변 Jacobi SVD
- **InPhase** — Daniel 2000 §3.30 위상 동기
**런타임 전환**: OSC `/sys/ambi_decoder_type i {0..4}`

### 3.3 VST3 플러그인 프로덕션 그레이드 (Phase C C2 Option B)
**무엇**: JUCE의 plugin client 의존성 없이 vst3sdk만으로 핸드롤한 VST3 플러그인.
**왜 중요**: DAW에 직접 로드 가능, X11/freetype/ALSA 등 sysdep 0건.
**파라미터 7종**:
1. Pan Azimuth, 2. Pan Elevation, 3. Source Width,
4. Master Gain, 5. Ambi Order, 6. Room Preset, 7. Bypass
**검증**:
- 1000-iter RT-safety probe (할당 0, malloc 오버라이드로 강제 검증)
- state v1↔v2 multi-version 리더 (v0.1.0 preset 자동 호환)
- 58/58 ctest PASS, .so 891KB
- 21-assertion 인-프로세스 호스트 픽스처

### 3.4 OFF byte-baseline 게이트 (CI 인프라)
**무엇**: 매 CI 빌드 산출물의 바이트 해시와 심볼 해시를 핀(pin).
**왜 중요**: 의도치 않은 의존성·툴체인 드리프트를 자동 차단.
**구조**:
- 듀얼 게이트: `libspe_core.a` + `spatial_engine_core` 동시 핀
- GHA `ubuntu-24.04` (GLIBC 2.39, GCC 13.3.0) 재현 빌드
- 공개 re-pin 경로: `GITHUB_STEP_SUMMARY`에 후보 해시 echo (토큰 불필요)

### 3.5 한국어 매뉴얼
**무엇**: 설치 + 운용 가이드 두 권.
**구성**:
- 설치 매뉴얼 12장: 하드웨어 요건, 빌드, 트러블슈팅
- 운용 매뉴얼 15장: 라이브 워크플로, OSC 프로토콜 레퍼런스, 라이선스
**대상**: 시스템 관리자, 음향 엔지니어, 통합 담당자

## 4. v0.2.0 검증 결과 (이번 주 측정)

### 4.1 핵심 테스트 수
| 항목 | 결과 |
|---|---|

### 4.2 RT-safety (1000-iter alloc-zero 강한 invariant)
| 테스트 | 결과 |
|---|---|
| `vst3_dispatch_rt_safety` rt_safety_alloc_zero_1000iter | ✅ alloc_total == 0 |
| `vst3_bypass` 7_rt_safety_bypass_alloc_zero | ✅ |
| `vst3_state_persist` 15_concurrent_set_get_state_1000iter | ✅ |
| Negative controls (probe_observes_malloc / calloc) | ✅ probe 동작 검증 |

### 4.3 ADM-OSC Soak (60s 실 UDP 부하)
```
soak_adm_osc_flood: 64 obj × 1 kHz × 60s (real UDP path)
  total sent:    762,496 packets
  xruns:         0          ✅
  rejectCount:   0          ✅
  batch p50:     4.92 ms
  batch p99:     6.25 ms    ⚠ > 3.0 ms (OS jitter, codec OK)
  wall-clock:    60.18 sec  PASS
```
**해석**: 디코더 자체는 결함 없음. p99 6.25ms는 PREEMPT_DYNAMIC 일반 커널의
스케줄링 지터. 프로덕션 환경(PREEMPT_RT + SCHED_FIFO)에서 ≤3 ms 도달 예상.

### 4.4 실시간 렌더 (NullBackend, 10s)
| 설정 | 블록 수 | xruns | 비고 |
|---|---|---|---|
| `--rate 48000 --block 64` | 7,509 | **0** | 48 kHz 정상 처리 |
| `--rate 48000 --block 32` | 15,020 | **0** | 더 짧은 버퍼에서도 안정 |

### 4.5 VST3 산출물 크기 및 sysdep

|---|---|
| `.so` 파일 크기 | 891 KB |

## 8. 핵심 기술 스펙

- **언어/플랫폼**: C++20 core + Python 3.11+ UI/bridge, Linux Ubuntu 24.04
- **CI baseline**: GHA `ubuntu-24.04` (GLIBC 2.39, GCC 13.3.0, CMake 3.20+)
- **오디오 I/O**: Digigram ALP-Dante (PCIe) — 프로덕션 / NullBackend — CI/dev
- **블록 크기**: 64 frames @ 48 kHz (32-block까지 xruns=0 확인)
- **레이턴시 목표**: OSC 수신 → 출력 p99 ≤ 5 ms (PREEMPT_RT 환경)
- **공간화 알고리즘**: VBAP, WFS, DBAP, HOA(MaxRE/AllRAD/EPAD/InPhase)
- **최대 오브젝트**: 64 동시
- **리버브**: 16-line Hadamard FDN + IRConvReverb (WAV/SOFA)
- **바이노럴**: KEMAR SOFA (RWTH Aachen, IR_len=384, 64800 positions)
- **IPC**: OSC/UDP, 포트 9100/9101, 10Hz 하트비트, 300ms 미스 게이트
- **UI**: PySide6 탑다운 뷰, 드래그 코알레서 120Hz, 매트릭스 뷰
- **VST3**: JUCE-free vst3sdk 핸드롤, .so 891KB, 7 파라미터, state v2
